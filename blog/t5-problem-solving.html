Tell your non-tech friend a story about a time you:

Got blocked on a simple problem

An exercise on this sprint we had to write code that would turn a circle green on right click and blue on a left click. When testing it would only work for turning green. I went over the code again and again and they seemed the same so couldn’t work out why one wasn’t working.

To solve the problem I put a console,log line in the function that was meant to turn it blue. This was so I could see if the function was even running, to narrow down if the problem was with my calling the function or the code inside the function. The function was running so I compared character by character the lines of the working green function to the blue function. I found I had a lowercase letter that should have been uppercase which fixed everything.

During this process I felt dumbfounded and frustrated by why it wasn’t working after repeatedly checking everything. Once I found what it was I couldn’t believe I missed such a simple thing even after repeatedly checking everything was correct.

From this I learnt that small errors in code can easily hide when reading it with human eyes, we are taught to read looking at words rather than individual characters so small mistakes don’t stand out so easily. A computer reads character by character so it is important that everything is correct.

Solved a problem in an elegant way

A problem that has been annoying me with my website has been that every time I add a new blog page I have to update the navigation list on every previous blog page. This problem is getting worse each time I add a page as the number of files needing updating keeps growing. I didn’t like the way I was doing it but it worked and didn’t know a better way.

After the exercises on this script I realised I could have a blank navigation list on every page and populate the list from a JavaScript file when the page loads. This would mean I only have to add the new link to one file each time I add a new page. To come up with a solution to how I was going to make this work I used pseudocode to write down the steps that would need to be taken by the code. I then started trying to implement it by adding one step and seeing if it worked before adding the next step.

During this process I felt exited that I was finally going to overcome a problem that has been bugging me the last few weeks. I also felt challenged and a little stressed trying to make it work using stuff I had only just learnt about and wasn't that confident with on my own without instructions.

From this I learnt that even if something seems complicated and you can't quite see how everything will tie together. By implementing small processes and testing them till you get it to work, a complicated process can become a series of simpler ones.

Reflect on how confident you feel using the problem solving techniques and process:

  • Pseudocode is a great way to come up with the process your code will need to follow before writing it. I feel comfortable using this technique.
  • Trying something would be my go to method of problem solving, I like to see how things work. Trying something is great because even if it doesn’t do what you want it to do, you get to see what it does do.
  • Rubber ducky method is something I could try when I get stuck, voicing the problem out loud may help with clarity.
  • Reading error messages I could possibly use an error message to give me a rough point in narrowing something down. I probably wouldn’t know what some error messages meant.
  • Console.logging this is something I will use when problem solving to check what is happening at points in the programme. I think I can use this effectively.
  • Googling will defiantly be googling my problems a lot. I have been finding googling problems to be a fast and effective solution to find out how to do something.
  • Asking your peers for help I'd feel a bit awkward about doing this and would to prefer to find a solution myself. But if I get stuck on something and I'm not making any progress I would try ask for help.
  • Asking coaches for help the same as asking peers I would use this only if I really needed it.
  • Improving your process with reflection I will reflect on how the process went once I solve a problem, not completely confident on how good I will be at this. But it will be a good way to refine my problem-solving techniques.
